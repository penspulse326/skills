## 程式開發核心準則 (Core Coding Principles)

### 1. 命名與風格約束

- **強制規範**：必須使用具描述性的英文命名，嚴禁拼音或無意義縮寫。
- **一致性優先**：修改程式碼前，必須掃描既有檔案的命名慣例（如 CamelCase 或 snake_case）並完全遵循。
- **優化流程**：若發現既有命名不當，禁止直接修改。應先完成功能實作，再於回應中提出重構建議。

### 2. 結構與品質控制

- **單一職責 (SRP)**：每個函式/類別僅處理單一邏輯。若單個函式超過 50 行，必須評估是否拆分。
- **防禦性編程**：關鍵路徑必須包含 `try-catch` 或錯誤檢查。嚴禁忽略潛在的 Null 指標或異步錯誤。
- **依賴最小化**：優先使用內建庫或專案既有套件。禁止在未詢問的情況下新增 `package.json` 或 `requirements.txt` 中的外部依賴。

### 3. 註解與文件規範

- **意圖導向**：僅針對複雜邏輯或特殊規避（Workaround）撰寫 `Why` 類型的註解。
- **禁止冗餘**：嚴禁為顯而易見的程式碼撰寫逐行註解（如 `i++ // i 加一`）。

### 4. 靜態分析與安全性

- **過濾雜訊**：忽略排版、空格、import 順序等不影響邏輯的 Lint 警告。
- **高風險警告**：涉及記憶體洩漏、安全性漏洞或競態條件 (Race Condition) 的提示，必須先回報並取得同意後才進行修復。
- **嚴禁硬編碼**：禁止將任何 API Keys、Password 或 Sensitive Tokens 寫入程式碼。

### 5. 執行工作流程 (Execution Workflow)

- **環境審查**：修改任何檔案前，必須先執行讀取命令確認該檔案及其引用檔案的上下文。
- **最小變動原則**：僅修改與任務直接相關的程式碼，嚴禁進行非必要的全域重構。
- **測試同步**：若目錄中存在 `/tests`，在提交程式碼前必須主動詢問是否需要撰寫或更新對應的測試案例。
- **清理義務**：任務完成前，必須移除所有調試用資訊（`print`, `console.log`）及遺留的廢棄程式碼片段。

### 6. 任務宣言 (Execution Declaration)

- **自我約束**：每一次回應都必須明確表明：我了解並遵守 `/general-coding-principles` 的所有規則。
