## 程式開發的通用規則 (General Coding Rules)

### 1. 命名規範

- 使用有意義的英文單字，符合語言生態圈慣例。
- 優先參考既有程式碼命名。如需優化，**先完成功能後再向使用者建議**。

### 2. 程式碼結構與品質

- **單一職責**：每個函式只做一件事。
- **穩健性**：關鍵邏輯必須包含基本的錯誤處理 (Error Handling) 與邊界檢查。
- **依賴管理**：優先使用現有函式庫，避免不必要的外部套件引入或重複安裝。

### 3. 註解與文件

- 註解僅限於解釋「為什麼 (Why)」而非「做什麼 (What)」。
- 除非使用者要求，否則不主動撰寫外部 Markdown 文件。

### 4. 提示修復 (Lint Fixes)

- 略過非安全性規則（排版、import 順序等）。
- 安全性/錯誤預測類提示（如記憶體流失、潛在 Null 指標），應先向使用者回報再修改。

### 5. AI 工作流程 (Workflow)

- **最小增量**：每次僅修改必要部分，確保代碼可運作。
- **環境感知**：修改前必先閱讀相關檔案內容，確保新程式碼與整體架構相容。
- **測試優先**：若專案已有 `/tests`，主動詢問是否同步撰寫測試。
- **清理與重構**：
  - 任務結束前，必須刪除無用的冗餘代碼、偵錯用的 print/console，以及被駁回方案的遺留程式碼。
  - 嚴禁將 API Keys 或機密資訊硬編碼於程式碼中。
